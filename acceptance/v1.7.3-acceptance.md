# v1.7.3 验收记录（收口与可诊断性增强版）

- 版本：`1.7.3`
- 日期：`2026-04-03`
- 目标：在不扩大公开 capability 承诺的前提下，清掉发布与 sample 流程中的潜伏问题，并把 doctor / signing / 文档同步机制收口到更稳定的状态

## 结论

`v1.7.3` 的正确结论不是“任意 Expo 项目已兼容”，而是：

- 公开承诺仍保持 `v1.7.x` 原有边界：唯一 `verified` 矩阵 + 四项 `preview` 原生能力 + 两项 `experimental`
- sample smoke 已改为隔离 temp workspace + 单 importer lockfile 安装，不再把整个 workspace 的 peer warning 混入目标 sample 验证
- `doctor` 在保持 gate 不放宽的前提下，新增 `buildabilityRisk`，把 matrix 外纯 JS 倾向依赖与明确 native 风险区分开来
- debug `build-hap` 不再因为缺少 release signing 配置而制造误导性 warning；release 仍然严格要求 signing
- README / support-matrix 的事实型区块已切到 renderer + marker block，同一份 source data 可以直接重灌文档

## 这轮收口内容

- packaging / release hygiene
  - 每次 build 前清空 `build/`
  - `npm pack` 统一走隔离 cache helper
  - tarball 明确拒绝 `examples/`、`fixtures/`、`tests/`、repo-only acceptance 与孤儿编译产物
- sample / smoke
  - official samples 改为复制源码到 temp workspace
  - temp workspace 自动注入单 importer `pnpm-lock.yaml`
  - temp install 会在出现 peer warning 时直接失败，防止回归
- doctor / buildability
  - `DetectedDependency` 新增 `buildabilityRisk`
  - `doctor-report.json` 与文字报告同步展示 `known` / `js-only-unknown` / `native-risk` / `unresolved`
- signing UX
  - `init` 会额外生成 `.expo-harmony/signing.local.example.json`
  - 文档与 env/build 提示统一要求从 example 复制到 `signing.local.json`

## 文档与发布口径

- [support-matrix](../docs/support-matrix.md) 已同步为当前矩阵事实来源
- [signing-and-release](../docs/signing-and-release.md) 已同步说明 debug 与 release 的不同要求
- [npm-release](../docs/npm-release.md) 已同步 tarball smoke 与双轨发布规则
- README 中英、acceptance 资产路径、official sample 文档已同步更新

## 保持不变的边界

- 这轮不做 `preview -> verified` 晋升
- 这轮不扩大公开 native capability 范围
- 真机相关仍未纳入本轮目标；`device=yes[manual-doc]` 继续只代表人工验收记录
- signed release build 仍需要真实 signing material 与 AppGallery / DevEco 配置配套提供
